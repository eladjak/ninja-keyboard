## Root cause of the last failing test (was misdiagnosed as a timing flake)

The test `tests/e2e/lessons-flow.spec.ts › results modal shows after lesson
completion` was NOT a dev-server hydration-latency flake. The real cause:

1. The test injected the completed-lesson value into `localStorage` via
   `page.evaluate` **after** navigating to `/lessons/lesson-01`.
2. That lesson page already had a live zustand `persist` xp-store mounted with an
   empty default. ~0.5s after the injection, the store flushed its in-memory
   (empty) state back to `localStorage`, **clobbering** the injected value.
3. By the time the test navigated to `/lessons` and asserted `'10 מ/ד'`, the
   persisted value was gone — so the text never rendered, even with a 15s timeout
   (confirmed: a production build alone did NOT fix it).

## The fix (commit `79d7da3`)

1. **Seed before app code runs.** Inject the persisted xp-store state via
   `page.addInitScript` (helper `seedCompletedLessonOne`) so the value is present
   in `localStorage` *before* the store first rehydrates. The live store then
   rehydrates **with** the completed lesson and never overwrites it. Applied to
   both data-dependent tests (`results modal…` and `completing first lesson
   unlocks second lesson`). Verified deterministic locally: 6/6 with
   `--retries=0` against a production server, and the full `lessons-flow` spec
   17/17 on chromium.
2. **Production-build E2E (robustness).** The CI `webServer` now runs
   `next build && next start` instead of `next dev` (gated on `CI`), removing the
   on-demand dev-server compile latency that drove the broader flaky-test class.
   Local dev still uses `next dev` with `reuseExistingServer`.
